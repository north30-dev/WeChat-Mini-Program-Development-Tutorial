## 第21章 云开发方案

微信云开发是微信官方提供的一站式后端云服务，开发者无需搭建服务器即可使用数据库、云函数和云存储。对于 Java 开发者来说，理解云开发有助于在项目初期快速验证想法，或在资源有限时降低运维成本。

### 21.1 云开发 vs 自建Java后端

| 对比维度 | 微信云开发 | 自建Java后端 |
|----------|-----------|-------------|
| **服务器** | 无需购买和维护 | 需购买云服务器（ECS/CVM） |
| **数据库** | 云数据库（MongoDB风格） | MySQL/PostgreSQL（关系型） |
| **API** | 云函数（Node.js） | Spring Boot REST API |
| **文件存储** | 云存储（类似OSS） | OSS/COS 自建 |
| **登录鉴权** | 内置，自动获取openid | 需自行实现code2Session |
| **开发语言** | JavaScript/TypeScript | Java（也可用Kotlin） |
| **运维成本** | 低（免运维） | 高（需运维监控） |
| **性能上限** | 受云函数配额限制 | 可水平扩展 |
| **成本** | 免费额度 + 按量付费 | 固定服务器费用 |
| **适用场景** | 小型项目、MVP验证 | 中大型项目、复杂业务 |
| **团队技能** | 前端友好 | Java后端友好 |

> **类比理解**：云开发类似于 Java 世界中的 **Serverless（如阿里云函数计算、AWS Lambda）**——你只需写业务逻辑函数，不需要关心服务器、数据库部署和运维。但它也有局限性：不支持长连接（WebSocket 需特殊处理）、冷启动延迟、云函数执行时间限制（默认3秒，最长60秒）等。

### 21.2 云数据库

云数据库采用 MongoDB 风格的 JSON 文档存储，与关系型数据库有显著差异：

```javascript
// 云开发初始化（app.js）
wx.cloud.init({
  env: 'your-env-id',
  traceUser: true
})

// 在页面中操作数据库
const db = wx.cloud.database()

// 查询文章列表（分页）
Page({
  async loadArticles() {
    const { data } = await db.collection('articles')
      .where({ status: 1 })
      .orderBy('publishedAt', 'desc')
      .skip((this.data.page - 1) * 10)
      .limit(10)
      .get()

    this.setData({
      articles: this.data.articles.concat(data)
    })
  },

  // 添加收藏
  async addFavorite(articleId) {
    const { result } = await wx.cloud.callFunction({
      name: 'favorite',
      data: { action: 'add', articleId }
    })
  }
})
```

> **Java开发者注意点**：云数据库的 JSON 文档模型与 MySQL 的关系模型有本质区别：
>
> - **无表结构定义**：不需要提前建表，直接插入文档即可。类似于 Java 中使用 `Map<String, Object>` 而非强类型实体类。
> - **无JOIN查询**：不支持表关联查询，需要通过冗余字段或多次查询实现。MySQL 中 `SELECT a.*, c.name FROM articles a JOIN categories c ON a.category_id = c.id` 在云数据库中需要先查文章列表，再根据 `categoryId` 批量查询分类。
> - **弱事务支持**：云数据库不支持多文档事务（仅单文档事务），而 MySQL 的 `@Transactional` 可以保证跨表事务一致性。
>
> 对于习惯了 JPA/MyBatis 的 Java 开发者，云数据库的灵活性反而可能带来数据一致性的隐患。

### 21.3 云函数

云函数运行在 Node.js 环境中，与 Java Spring Boot 的 Controller + Service 体系对比：

```javascript
// 云函数：favorite/index.js
const cloud = require('wx-server-sdk')
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })
const db = cloud.database()

// 云函数入口函数
exports.main = async (event, context) => {
  const { OPENID } = cloud.getWXContext()
  const { action, articleId } = event

  switch (action) {
    case 'add':
      // 检查是否已收藏
      const { data: existing } = await db.collection('favorites')
        .where({ userId: OPENID, articleId })
        .get()
      if (existing.length > 0) return { code: 0, message: '已收藏' }

      await db.collection('favorites').add({
        data: {
          userId: OPENID,
          articleId,
          createdAt: db.serverDate()
        }
      })
      return { code: 0, message: '收藏成功' }

    case 'remove':
      await db.collection('favorites')
        .where({ userId: OPENID, articleId })
        .remove()
      return { code: 0, message: '已取消收藏' }

    case 'check':
      const { data: fav } = await db.collection('favorites')
        .where({ userId: OPENID, articleId })
        .get()
      return { code: 0, data: fav.length > 0 }

    default:
      return { code: -1, message: '未知操作' }
  }
}
```

> **类比理解**：云函数的 `exports.main` 相当于 Spring Boot 中 `@PostMapping("/api/favorites")` 的 Controller 方法。`event` 参数相当于 `@RequestBody`，`context` 包含运行环境信息。`cloud.getWXContext().OPENID` 自动获取用户身份，无需像 Java 后端那样手动解析 JWT。

### 21.4 云存储

```javascript
// 上传图片到云存储
wx.cloud.uploadFile({
  cloudPath: `images/${Date.now()}-${Math.random().toString(36).substr(2)}.${ext}`,
  filePath: tempFilePath,  // 本地临时文件路径
  success: (res) => {
    console.log('上传成功，fileID：', res.fileID)
    // fileID 格式：cloud://your-env-id.xxxx/images/xxx.jpg
  }
})

// 获取临时下载链接
wx.cloud.getTempFileURL({
  fileList: ['cloud://your-env-id.xxxx/images/xxx.jpg'],
  success: (res) => {
    console.log('临时链接：', res.fileList[0].tempFileURL)
  }
})
```

> **类比理解**：云存储类似于阿里云 OSS 或腾讯云 COS。`cloudPath` 相当于 OSS 的 Object Key，`fileID` 是云存储的唯一标识。与 Java 后端中使用 OSS SDK 上传文件的流程一致，只是调用方式从 Java SDK 变成了小程序 API。

### 21.5 选型建议

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 个人项目 / 毕业设计 | 云开发 | 零运维，快速上线 |
| MVP 验证 | 云开发 | 快速验证，低成本试错 |
| 已有Java后端团队 | 自建Java后端 | 复用现有技术栈和经验 |
| 复杂业务逻辑 | 自建Java后端 | 事务、定时任务、消息队列等 |
| 高并发场景 | 自建Java后端 | 可水平扩展，性能可控 |
| 内容管理后台 | 自建Java后端 | 需要管理端+小程序端共享数据 |
| 纯展示类小程序 | 云开发 | 数据量小，逻辑简单 |

> **Java开发者注意点**：如果你所在团队已有成熟的 Spring Boot 技术栈和运维体系，建议选择自建后端——可以复用现有的用户体系、权限管理、日志监控等基础设施。云开发更适合前端工程师独立完成全栈开发，或快速验证产品想法的场景。
