## 第26章 性能优化与安全规范

### 26.1 包体积优化

微信小程序对包体积有严格限制：

| 限制项 | 大小 |
|--------|------|
| 主包 | 不超过 2MB |
| 单个分包 | 不超过 2MB |
| 整体（含分包） | 不超过 20MB |

**`app.json` 分包配置示例**：

```json
{
  "pages": [
    "pages/index/index",
    "pages/profile/profile"
  ],
  "subpackages": [
    {
      "root": "packageA",
      "name": "shop",
      "pages": [
        "pages/product/list",
        "pages/product/detail",
        "pages/cart/index",
        "pages/order/confirm"
      ]
    },
    {
      "root": "packageB",
      "name": "community",
      "pages": [
        "pages/post/list",
        "pages/post/detail",
        "pages/post/create"
      ]
    }
  ],
  "preloadRule": {
    "pages/index/index": {
      "network": "wifi",
      "packages": ["packageA"]
    }
  }
}
```

**分包优化策略**：

1. **独立分包**：分包可独立运行，不依赖主包，启动更快
2. **分包预下载**：在 `preloadRule` 中配置，进入某些页面时预下载可能用到的分包
3. **按功能拆分**：将低频功能（如设置、关于、帮助）放入独立分包

```json
{
  "subpackages": [
    {
      "root": "packageA",
      "name": "shop",
      "independent": true,
      "pages": ["pages/product/list"]
    }
  ]
}
```

> **类比理解：** 分包策略类似于 Java Web 中的微服务拆分思想——将单体应用按功能边界拆分为多个独立模块，每个模块独立部署和加载。`preloadRule` 则类似于前端路由中的 prefetch 机制。

### 26.2 图片资源优化

| 优化手段 | 说明 | 效果 |
|---------|------|------|
| 图片压缩 | 使用 TinyPNG 等工具压缩 PNG/JPG | 体积减少 40%-70% |
| WebP 格式 | 使用 WebP 格式替代 PNG/JPG | 体积减少 25%-35% |
| CDN 加速 | 图片上传至 CDN，不在小程序包内 | 大幅减小包体积 |
| 懒加载 | 使用 `lazy-load` 属性延迟加载 | 减少首屏加载时间 |
| 按需加载 | 不同分辨率设备加载不同尺寸图片 | 节省带宽 |

```xml
<!-- 图片懒加载示例 -->
<image lazy-load src="{{item.imageUrl}}" mode="aspectFill" />
```

```javascript
// 根据设备像素比加载不同尺寸图片
const systemInfo = wx.getSystemInfoSync()
const pixelRatio = systemInfo.pixelRatio
// pixelRatio 为 2 时加载 2x 图，为 3 时加载 3x 图
const imageUrl = `https://cdn.example.com/images/banner@${pixelRatio}x.webp`
```

### 26.3 setData 性能优化

`setData` 是小程序中最容易引发性能问题的 API。它的工作原理是将数据从逻辑层（JavaScript Core）通过序列化传输到渲染层（WebView），因此每次调用都有序列化和通信开销。

**优化原则与示例**：

```javascript
// ❌ 错误：频繁调用 setData
for (let i = 0; i < list.length; i++) {
  this.setData({ [`list[${i}].loaded`]: true })  // 触发 N 次通信
}

// ✅ 正确：合并为一次 setData
const updates = {}
for (let i = 0; i < list.length; i++) {
  updates[`list[${i}].loaded`] = true
}
this.setData(updates)  // 只触发 1 次通信
```

```javascript
// ❌ 错误：传递大量不需要渲染的数据
this.setData({
  list: hugeArray,        // 1000 条数据全部传输到渲染层
  userInfo: userData,
  config: appConfig
})

// ✅ 正确：只传递渲染所需的数据
this.setData({
  'list': hugeArray.slice(0, 20),  // 只传当前页显示的数据
  'userInfo.name': userData.name,   // 使用路径表达式精确更新
  'userInfo.avatar': userData.avatar
})
```

```javascript
// ❌ 错误：setData 中包含不需要渲染的字段
this.setData({
  list: newList,
  _internalCounter: counter,  // 下划线开头的内部变量不需要渲染
  _cacheData: cache
})

// ✅ 正确：只更新与视图绑定的数据
this.setData({ list: newList })
// 内部变量直接赋值，不通过 setData
this._internalCounter = counter
this._cacheData = cache
```

> **Java 开发者注意点：** `setData` 的性能问题类似于 Java Web 中频繁的 DOM 操作。在传统 Web 开发中，我们会使用虚拟 DOM（React/Vue）来批量合并更新。小程序没有虚拟 DOM，因此需要开发者手动控制 `setData` 的调用频率和数据量。核心原则是：**减少调用次数、减少传输数据量、使用路径表达式精确更新**。

### 26.4 列表渲染优化

```xml
<!-- ✅ 必须指定 wx:key -->
<block wx:for="{{list}}" wx:key="id">
  <view class="item">{{item.name}}</view>
</block>

<!-- ❌ 不要使用 index 作为 key（列表会变动时） -->
<block wx:for="{{list}}" wx:key="index">
  <view class="item">{{item.name}}</view>
</block>
```

**虚拟列表方案**（长列表场景）：

```javascript
// 使用 recycle-view 组件（微信官方提供）
// npm install --save miniprogram-recycle-view

// pages/list/list.json
{
  "usingComponents": {
    "recycle-view": "miniprogram-recycle-view/recycle-view",
    "recycle-item": "miniprogram-recycle-view/recycle-item"
  }
}
```

```xml
<!-- pages/list/list.wxml -->
<recycle-view id="recycleId" batch="{{batchSetRecycleData}}" height="{{height}}">
  <view slot="before">头部固定区域</view>
  <recycle-item wx:for="{{recycleList}}" wx:key="id">
    <view class="item">
      <text>{{item.title}}</text>
      <text>{{item.desc}}</text>
    </view>
  </recycle-item>
  <view slot="after">底部固定区域</view>
</recycle-view>
```

### 26.5 安全规范

#### HTTPS 强制

微信小程序 **强制要求** 所有网络请求使用 HTTPS。开发时可在开发者工具中勾选"不校验合法域名"进行调试，但正式环境必须配置合法域名。

```
配置路径：微信公众平台 → 开发管理 → 开发设置 → 服务器域名
- request 合法域名：https://api.example.com
- socket 合法域名：wss://ws.example.com
- uploadFile 合法域名：https://upload.example.com
- downloadFile 合法域名：https://download.example.com
```

> **Java 开发者注意点：** 这与 Spring Boot 中强制 HTTPS 的做法一致。在 Java 中通常通过配置 `server.ssl.enabled=true` 或 Nginx 反向代理配置 SSL 证书来实现。小程序更进一步，直接在平台层面强制校验，无法绕过。

#### 数据加密

```javascript
// 敏感数据加密传输（与 Java 后端对接）
const crypto = require('../../utils/crypto')

// 发送加密请求
const encryptedData = crypto.aesEncrypt(JSON.stringify(sensitiveData), secretKey)
wx.request({
  url: 'https://api.example.com/secure/submit',
  method: 'POST',
  data: { encrypted: encryptedData },
  header: { 'Content-Type': 'application/json' }
})
```


#### 防 XSS（跨站脚本攻击）

小程序的 WXML 模板引擎 **默认转义** 输出内容，天然具备一定的 XSS 防护能力：

```xml
<!-- 以下两种写法都是安全的，小程序会自动转义 -->
<text>{{userInput}}</text>
<view>{{htmlContent}}</view>

<!-- ⚠️ 使用 rich-text 组件渲染 HTML 时需注意 -->
<rich-text nodes="{{trustedHtml}}"></rich-text>
<!-- trustedHtml 必须经过服务端白名单过滤，切勿直接渲染用户输入 -->
```

#### 防 CSRF（跨站请求伪造）

小程序运行在微信客户端沙箱中，**天然免疫传统 CSRF 攻击**（因为不存在跨域 Cookie 自动携带的场景）。但建议仍采用以下最佳实践：

- 使用 `wx.login()` 获取的 `code` 换取 `openid` 和 `session_key`，不依赖 Cookie
- 使用自定义的 `token`（存储在 `Storage` 中）进行接口鉴权
- 敏感操作增加短信验证码或支付密码二次验证

### 26.6 敏感信息保护

```
绝对不能暴露给前端的信息：
├── AppSecret（仅存放在后端服务器）
├── 数据库连接字符串
├── 第三方服务密钥（支付密钥、短信密钥等）
├── 后端服务内网地址
└── 业务核心算法逻辑
```

```javascript
// ❌ 错误：在前端代码中硬编码密钥
const APP_SECRET = 'your_app_secret_here'  // 绝对禁止！

// ✅ 正确：敏感操作全部在后端完成
// 前端只调用后端接口，后端使用 AppSecret 完成签名等操作
wx.request({
  url: 'https://api.example.com/payment/create',
  method: 'POST',
  data: { orderId: '12345' },
  header: { 'Authorization': 'Bearer ' + token }
})
```

> **Java 开发者注意点：** 这与 Spring Boot 安全最佳实践完全一致——敏感配置通过环境变量或配置中心注入，绝不硬编码在代码中。小程序前端代码虽然经过编译混淆，但仍可被反编译，因此更不能在前端存放任何密钥或敏感信息。

### 26.7 与 Java Web 安全规范对比

| 安全领域 | 微信小程序 | Java Web（Spring Security） |
|---------|-----------|---------------------------|
| 传输加密 | 强制 HTTPS | 推荐 HTTPS，需自行配置 |
| 身份认证 | `wx.login()` + 自定义 token | Spring Security / JWT / OAuth2 |
| XSS 防护 | 模板引擎默认转义 | Thymeleaf 默认转义 / XSS Filter |
| CSRF 防护 | 沙箱天然免疫 | CSRF Token / SameSite Cookie |
| 数据加密 | 前端加密 + 后端解密 | HTTPS + AES/RSA 加密 |
| 接口限流 | 后端实现（与 Java 一致） | Spring Cloud Gateway / Sentinel |
| 权限控制 | 后端 RBAC（与 Java 一致） | Spring Security RBAC |

---

> **本部分小结：** 微信小程序的发布运维与 Java Web 应用既有相似之处，也有其独特性。版本管理、多环境配置、性能优化等理念是相通的，而审核机制、平台级监控、包体积限制等则是小程序生态特有的。掌握这些知识后，你就能将 Java Web 开发中的工程化经验平滑迁移到小程序开发中，构建高质量的小程序应用。
