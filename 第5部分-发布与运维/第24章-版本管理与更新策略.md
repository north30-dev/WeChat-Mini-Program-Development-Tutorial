## 第24章 版本管理与更新策略

### 24.1 语义化版本规范

微信小程序推荐使用 **语义化版本号**（Semantic Versioning）：

```
格式：主版本号.次版本号.修订号
示例：1.2.3

- 主版本号（Major）：不兼容的 API 变更，如底层架构重构
- 次版本号（Minor）：向下兼容的功能新增，如新增页面或接口
- 修订号（Patch）：向下兼容的问题修复，如 Bug 修复
```

> **类比理解：** 这与 Maven/Gradle 中的版本号规范完全一致。`1.2.3` 在 Java 世界里同样表示"第 1 个大版本、第 2 次功能迭代、第 3 次 Bug 修复"。

### 24.2 版本更新检测与强制更新

在 `app.js` 的 `onLaunch` 中检测版本更新，提示用户升级：

```javascript
// app.js
App({
  onLaunch() {
    this.checkUpdate()
  },

  checkUpdate() {
    if (wx.canIUse('getUpdateManager')) {
      const updateManager = wx.getUpdateManager()

      updateManager.onCheckForUpdate(function (res) {
        // res.hasUpdate 为 true 时表示有新版本
        console.log('是否有新版本：', res.hasUpdate)
      })

      updateManager.onUpdateReady(function () {
        wx.showModal({
          title: '更新提示',
          content: '新版本已经准备好，是否重启应用？',
          success(res) {
            if (res.confirm) {
              // 强制重启并应用新版本
              updateManager.applyUpdate()
            }
          }
        })
      })

      updateManager.onUpdateFailed(function () {
        // 新版本下载失败
        wx.showModal({
          title: '更新提示',
          content: '新版本下载失败，请检查网络后重试',
          showCancel: false
        })
      })
    }
  }
})
```

**服务端版本控制方案**（适用于强制更新）：

```javascript
// 在 app.js 中请求后端接口检查最低支持版本
async checkForceUpdate() {
  const res = await wx.request({
    url: 'https://api.example.com/version/check',
    method: 'GET'
  })
  const { minVersion, latestVersion, updateUrl } = res.data

  const currentVersion = this.globalData.version // 从全局配置获取当前版本

  if (this.compareVersion(currentVersion, minVersion) < 0) {
    // 当前版本低于最低支持版本，强制更新
    wx.showModal({
      title: '版本过低',
      content: '当前版本过低，请升级到最新版本后使用',
      showCancel: false,
      success() {
        if (updateUrl) {
          wx.setClipboardData({ data: updateUrl })
        }
      }
    })
  }
},

// 版本号比较工具函数
compareVersion(v1, v2) {
  const v1Parts = v1.split('.')
  const v2Parts = v2.split('.')
  const len = Math.max(v1Parts.length, v2Parts.length)

  for (let i = 0; i < len; i++) {
    const num1 = parseInt(v1Parts[i] || 0)
    const num2 = parseInt(v2Parts[i] || 0)
    if (num1 > num2) return 1
    if (num1 < num2) return -1
  }
  return 0
}
```

### 24.3 热更新机制

微信小程序 **不支持** 传统意义上的热更新（即绕过审核直接更新代码）。所有代码变更必须经过上传 → 审核 → 发布的完整流程。

| 对比项 | 微信小程序 | Java Web |
|--------|-----------|----------|
| 代码更新方式 | 上传 → 审核 → 发布 | 直接部署新版本 |
| 热更新支持 | 不支持（仅支持配置热更新） | 支持（热部署、热加载） |
| 配置更新 | 可通过云开发数据库或后端接口动态下发 | 通过配置中心（Nacos/Apollo）实现 |
| 生效时间 | 用户冷启动时检测 | 取决于部署策略 |

> **Java 开发者注意点：** 在 Spring Boot 开发中，我们习惯使用 Spring DevTools 实现热重载，或通过 JRebel 实现热部署。小程序没有类似的机制，但可以通过以下方式实现"伪热更新"：
> - **配置热更新**：将 UI 文案、开关配置等存放在后端接口或云数据库中，前端启动时拉取
> - **业务规则热更新**：将业务逻辑放在服务端，前端只负责展示

### 24.4 版本回滚操作

在微信公众平台 → **版本管理** → **线上版本** 中：

1. 找到需要回滚的历史版本
2. 点击 **"回退"** 按钮
3. 确认回退操作

> **注意：** 回退操作会将线上版本回退到上一个已发布的版本，且 **24 小时内只能回退一次**。

### 24.5 三套环境管理

| 环境 | 说明 | 访问方式 |
|------|------|---------|
| 开发版 | 开发者工具中实时预览，仅开发者可访问 | 开发者工具编译 |
| 体验版 | 上传后未审核的版本，通过二维码分享给测试人员 | 体验版二维码 |
| 正式版 | 审核通过并发布的版本，所有用户可访问 | 微信搜索/扫码 |

**环境切换配置示例**：

```javascript
// config/env.js
const ENV = {
  development: {
    baseUrl: 'https://dev-api.example.com',
    wsUrl: 'wss://dev-ws.example.com',
    staticUrl: 'https://dev-static.example.com'
  },
  staging: {
    baseUrl: 'https://staging-api.example.com',
    wsUrl: 'wss://staging-ws.example.com',
    staticUrl: 'https://staging-static.example.com'
  },
  production: {
    baseUrl: 'https://api.example.com',
    wsUrl: 'wss://ws.example.com',
    staticUrl: 'https://static.example.com'
  }
}

// 根据小程序环境自动选择
const accountInfo = wx.getAccountInfoSync()
const envVersion = accountInfo.miniProgram.envVersion
// envVersion: 'develop' | 'trial' | 'release'

const currentEnv = ENV[envVersion === 'release' ? 'production' : envVersion === 'trial' ? 'staging' : 'development']

module.exports = { currentEnv }
```

> **类比理解：** 这与 Spring Boot 中通过 `spring.profiles.active` 切换 `application-dev.yml`、`application-staging.yml`、`application-prod.yml` 的多环境配置思路完全一致。
